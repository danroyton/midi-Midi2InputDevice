# ADR-002: MidiMapper 作为 Windows 服务

| 字段 | 值 |
|---|---|
| 状态 | **已接受** |
| 日期 | 2025-05-30 |
| 决策者 | 开发团队 |

## 背景

后端可以作为**普通桌面应用程序**（托盘程序）或 **Windows 服务**运行。两种方式在延迟、自动启动、安全性以及注入键盘输入的能力方面有不同的影响。

## 问题：SendInput 与 Windows 服务

Windows 服务默认运行在 **Session 0**（隔离，非交互式）。`SendInput` 和类似的 UI 自动化 API **仅在交互式用户会话**（Session 1+）中有效。

解决方案：

| 方案 | 实现难度 | 可靠性 |
|---|---|---|
| 服务 + 在 Session 1 中通过 `CreateProcessAsUser` 启动的托盘辅助进程 | 高 | 高 |
| 服务使用 `AllowInteractToDesktop`（已过时，存在安全风险） | 低 | 低 |
| 无服务；通过任务计划程序（交互式）自动启动的普通进程 | 低 | 中 |
| 服务通过命名管道向托盘进程发送注入命令 | 中 | 高 |

## 选项

### 选项 A：无服务——通过任务计划程序自动启动
- 后端进程在用户登录时启动。
- 运行于用户会话中 → `SendInput` 可直接使用。
- **优点：** 简单，无 Session 0 问题。
- **缺点：** 无法在未登录用户的情况下运行。
- **缺点：** 崩溃后无自动重启（需额外配置）。

### 选项 B：Windows 服务 + 命名管道到托盘代理
- 服务管理 MIDI 输入、映射引擎、API。
- Session 1 中的最小托盘代理通过命名管道接收注入命令并调用 `SendInput`。
- **优点：** 自动启动，无需登录即可运行。
- **优点：** 服务逻辑与 UI 注入职责清晰分离。
- **缺点：** 两个进程，通信复杂度更高。

### 选项 C：Windows 服务，直接在交互式会话中注入
- 服务通过 `WTSGetActiveConsoleSessionId` 获取活动会话。
- 通过 `CreateProcessAsUser` 启动注入辅助进程。
- **优点：** 全自动，无需手动托盘代理。
- **缺点：** 实现复杂度高，需要提升权限。

## 决策

**两阶段方案：**
1. **阶段 1：** 选项 A（无服务，通过任务计划程序自动启动）。快速落地，无会话问题。
2. **阶段 2（可选）：** 选项 B——如有无用户登录运行的需求，切换至 Windows 服务 + 托盘代理。

架构从一开始就将注入逻辑封装在接口（`IInputInjector`）后面，无需更改其余逻辑即可切换为命名管道代理。

## 理由

对于主要使用场景（个人 PC 上的游戏/生产力），无需在无用户登录时运行服务。选项 B 的额外工作量只有在明确提出该需求时才值得。

## 影响

- 定义 `IInputInjector` 接口。
- 阶段 1 实现：在后端进程中直接调用 `SendInput`。
- `BackgroundService.ExecuteAsync` 在专用线程上以 `ThreadPriority.Highest` 执行。
- 安装程序/设置脚本中创建任务计划程序任务。
